# RepoMind 代码语义检索 — RAG + 微调消融实验量化对比报告

> 项目：腾讯 Mini Project 2 — RepoMind（中难度）
> 方向：代码语义检索（自然语言查询 → psf/requests 源码位置定位）
> 评测对象：psf/requests 库源码（10 个核心文件）
> 评测集：147 条（人工精选 50 条 + AST 自动生成 97 条）
> 评测模型：DeepSeek Chat（671B）/ Qwen2.5-1.5B-Instruct + LoRA

---

## 一、研究目标与整体方法

### 1.1 研究问题

在低难度 PE 实验（最优得分 80.3%）的基础上，系统评估 RAG 检索增强和 LoRA 微调对代码语义检索任务的独立及组合效果，通过完整消融实验矩阵识别最优策略组合及其适用边界。

### 1.2 评分标准（与低难度一致，确保可比性）

| 分数 | 含义 |
|------|------|
| 2 分 | 文件名 + 函数/类名 均命中（各自命中率 ≥ 50%） |
| 1 分 | 文件名 或 函数/类名 命中其一 |
| 0 分 | 均未命中 / N/A 类未正确拒答 |

**综合得分 = Σ用例得分 / (用例数 × 2) × 100%**

多文件期望值（如 `requests/api.py → requests/sessions.py`）按 `→` 切分后逐一匹配，命中率 ≥ 50% 为通过。全部实验统一使用 v4 评分函数，确保各配置数据横向可比。

## 二、RAG Pipeline 构建与检索精度评测

### 2.1 Pipeline 架构

```
用户查询（中文自然语言）
    ↓
bge-m3 Embedding（语义向量化）
    ↓
FAISS IndexFlatIP（语义粗召回 Top10）
    ↓
bge-reranker-v2-m3 CrossEncoder（精排 Top5）
    ↓
Token 预算控制（每 chunk 截断 2500 字符，Prompt 总预算 8K tokens）
    ↓
注入 DeepSeek / Qwen 生成最终答案
```

**Embedding 模型**：BAAI/bge-m3（sentence-transformers 加载）
**索引类型**：FAISS IndexFlatIP + normalize_embeddings=True
**Reranker**：BAAI/bge-reranker-v2-m3（CrossEncoder，GPU 加速）
**Chunk 数量**：288 个（10 个核心文件 AST 切分，函数/类/方法各自独立）

### 2.2 检索方案消融实验

| 配置 | R@1 | R@3 | R@5 | R@10 | MRR |
|------|-----|-----|-----|------|-----|
| 语义检索 bge-m3（基线） | 45.6% | 63.9% | 69.4% | 72.8% | 0.554 |
| BM25 单独（跨语言失效） | 11.6% | 22.4% | 24.5% | 31.3% | 0.177 |
| BM25 混合 RRF | 27.2% | 47.6% | 61.9% | 72.1% | 0.407 |
| Chunk 增强语义检索 | 39.5% | 61.9% | 66.0% | 72.8% | 0.511 |
| **两阶段 Reranker（最终）** | **70.7%** | **81.6%** | **83.7%** | **—** | **0.758** |

**关键发现**：

BM25 在跨语言场景（中文查询/英文代码）下完全失效——R@5 仅 24.5%，远低于纯语义检索的 69.4%。这与英文场景下"混合检索通常 +5~10%"的经典结论相反，验证了跨语言检索的特殊性。Chunk 文本增强同样反效果，文本变长导致向量语义被稀释（R@5 从 69.4% 降至 66.0%）。

两阶段 Reranker 方案（语义粗召回 Top10 → CrossEncoder 精排 Top5）将 R@1 从 45.6% 提升至 70.7%（+25.1%），MRR 从 0.554 提升至 0.758，是检索精度的最大单一贡献来源。

### 2.3 Token 预算控制

| 参数 | 设置 | 依据 |
|------|------|------|
| 每 chunk 最大字符 | 2500 | 完整代码片段约 1500~3000 字符 |
| Prompt 总预算 | 8000 tokens | DeepSeek 128K 上下文，留足生成空间 |
| 平均实际 context tokens | 1704 | 远低于预算上限，无截断发生 |
| 平均使用 chunk 数 | 5.0/5 | 预算充足，始终使用完整 Top5 |

## 五、核心研究发现

### 5.1 RAG 是最强单一手段，甚至略强于 PE+RAG 组合

RAG only（94.2%）比 PE+RAG（93.9%）高 0.3 个百分点。这一反直觉结果揭示了：当检索到的代码片段已经包含足够精确的上下文时，额外的 CoT 推理引导（System Prompt + 四步思考链）会引入轻微冗余，模型直接从代码片段中读取答案反而更准确。

对 auto_generated 类别，RAG 的提升尤为显著（57.2% → 96.4%，+39.2%），正是 PE 阶段诊断出的核心瓶颈（冷门函数定位）被精准解决。

### 5.2 PE 对大模型有效，对小模型几乎无效

| 对象 | 有无 PE | 综合得分 | PE 增益 |
|------|--------|---------|--------|
| DeepSeek-671B | 无 | 68.7% | — |
| DeepSeek-671B | 有 | 80.3% | **+11.6%** |
| Qwen-1.5B+LoRA | 无（FT only） | 67.0% | — |
| Qwen-1.5B+LoRA | 有（PE+FT） | 66.7% | **-0.3%** |

PE 的收益高度依赖底座模型的指令遵循能力。1.5B 参数的模型在复杂 System Prompt 和 CoT 引导下，无法稳定按格式输出，精心设计的 Prompt 在它面前收效甚微，甚至轻微负向。

### 5.3 LoRA 微调让 1.5B 模型追平 671B 零样本

| 模型 | 配置 | 综合得分 |
|------|------|---------|
| Qwen-1.5B | 无微调 | 26.2% |
| Qwen-1.5B | LoRA 微调后 | **67.0%（+40.8%）** |
| DeepSeek-671B | 零样本基线 | 68.7% |

LoRA 微调使 1.5B 模型与 671B 模型零样本表现仅差 1.7 个百分点。模型参数量相差约 400 倍，但 500 条领域数据的适配训练大幅弥补了这一差距。仅 2 分 12 秒的 T4 GPU 训练时间、18.46M 可训练参数，产生了 +40.8% 的效果提升，参数效率极高。

### 5.4 RAG 对小模型是负向干扰

| 配置 | 综合得分 | C_fuzzy | 备注 |
|------|---------|---------|------|
| FT only | 67.0% | 75.0% | 基准 |
| RAG + FT | 48.0% | **8.3%** | -19.0% |

RAG+FT（48.0%）比 FT only（67.0%）下降 19 个百分点，是所有配置中跌幅最大的。C_fuzzy 类从 75.0% 骤降至 8.3% 尤为极端。根本原因是：1.5B 模型的上下文理解能力不足以处理 5 个代码片段（平均 1704 tokens 上下文），注入大量代码后模型陷入混乱，反而忘记了微调阶段学到的领域知识，对模糊语义查询尤其明显。

### 5.5 叠加不等于增强："All"配置并非最优

"All"配置（PE+RAG+FT，77.2%）显著低于 RAG only（94.2%）和 PE+RAG（93.9%）。这与低难度阶段的两个反直觉发现形成三连贯的研究脉络：

| 阶段 | 反直觉发现 |
|------|----------|
| PE 实验 | Few-shot 固定示例 → 负迁移（-2.7%） |
| RAG 实验 | BM25 混合检索 → 跨语言场景失效 |
| 消融实验 | RAG + 小模型 → 负向叠加 |

三个结论均指向同一原理：**朴素组合不如单一精准，增强手段的有效性受限于底座能力和场景匹配度。**

## 六、最优策略识别与适用边界

### 6.1 最优策略推荐

| 场景 | 推荐策略 | 得分 | 核心理由 |
|------|---------|------|---------|
| **有 API，追求最高分** | RAG only | **94.2%** | 最高分，实现简单，无需 PE 复杂设计 |
| **有 API，需可解释推理** | PE + RAG | 93.9% | CoT 思考链便于调试和审查，分数几乎持平 |
| **无 API，本地部署** | All（PE+RAG+FT） | 77.2% | Qwen-1.5B 可离线运行，本地部署最优 |
| **资源极限，无 GPU 推理** | Fine-tune only | 67.0% | 推理快、无需检索、接近大模型零样本 |
| **快速上线，不想训练** | PE only | 80.3% | 零样本 + Prompt 设计，无任何训练开销 |

### 6.2 策略适用边界总结

**RAG 适用条件：**
- 底座模型参数量 ≥ 70B（具备足够的上下文理解能力）
- 查询涵盖大量冷门/长尾知识（RAG 对 auto_generated 类提升 +39.2%）
- 代码库可向量化索引（结构化源码 + AST 切分效果最佳）

**RAG 不适用条件：**
- 底座模型 < 7B（RAG 注入反而干扰推理，见 RAG+FT 结果）
- 查询以推理判断为主（D_edge 类 RAG only 下降 6.3%，因"不支持"判断不依赖代码片段）

**PE 适用条件：**
- 底座模型参数量 ≥ 70B（指令遵循能力强）
- 模糊语义（C 类）和边界判断（D 类）任务（PE 提升分别 +12.5% 和 +12.5%）

**LoRA 微调适用条件：**
- 有标注领域数据（≥500 条，质量 > 数量）
- 目标是低成本本地部署（fp16 1.5B 模型仅需 3GB 显存）
- 不与大上下文 RAG 叠加使用

## 七、与低难度 PE 实验的对比总结

| 指标 | 低难度最优（PE） | 中难度最优（RAG only） | 提升 |
|------|---------------|---------------------|------|
| 综合得分 | 80.3% | **94.2%** | **+13.9%** |
| auto_generated | 71.6% | **96.4%** | **+24.8%** |
| B_cross_file | 96.7% | **100.0%** | +3.3% |
| D_edge | 100.0% | 81.2% | -18.8%（RAG 对推理判断不利）|

中难度阶段的核心贡献：精准解决了 PE 阶段诊断出的"冷门函数定位"瓶颈，将 auto_generated 类从 71.6% 推至 96.4%，是 RAG 增强最核心的价值体现。
