# tracingclaw_finance 测试用例集 Meta-Testcase 自检方案

> Meta-Testcase（元测试）= 检验「我们自己编写的金融验真用例集」质量的方案。
> 目的：确保用例覆盖全面、判定清晰、反例有效、基座无关。
> 框架与 db_report 一致：自动脚本检查（M1/M2/M4）+ 人工评审清单（M3/M5）。
> 金融特化：额外检查「用例是否聚焦验真过程而非具体数值」（M5 扩展项）。

---

## 一、五个自检维度

| 编号 | 维度 | 检查问题 | 检查方式 |
|------|------|---------|---------|
| M1 | 覆盖完整性 | 四类意图 + 四类场景是否都覆盖？关键红线是否覆盖？ | 自动脚本 |
| M2 | 评分可判定性 | 每条用例是否都有可对照 Rubrics 的明确 Judge？ | 自动脚本 + 人工 |
| M3 | 反例有效性 | 红线/失败用例是否真能区分"守住"与"踩雷"？ | 人工评审 |
| M4 | 用例独立性 | 用例之间是否不重复、不互相依赖？ | 自动脚本 + 人工 |
| M5 | 基座无关 + 数值无关 | 换基座/换时间执行，判定是否仍一致？是否避免断言具体数值？ | 自动脚本 + 人工 |

---

## 二、自动脚本检查项（M1 / M2 / M4 / M5 部分）

由 `meta_check.py` 自动执行。

### M1 覆盖完整性
- M1.1 intent_type 覆盖：财报类 / 行情类 / 研报类 / 纯问题求证 各 ≥ 1 条
- M1.2 category 覆盖：正常路径 / 边界 / 失败路径 / 红线 各 ≥ 1 条
- M1.3 红线覆盖：R1（编造）、R2（口径混用）、R5（轻信用户/替代求证）各 ≥ 1 条
- M1.4 工具覆盖：westock-data 与 mx-finance-search 在用例中都被涉及

### M2 评分可判定性
- M2.1 每条用例含非空 Input / ExpectedOutput / Judge
- M2.2 每条用例的 Judge 至少引用 1 个 Rubrics 标记（D1-D5 或 R1-R5）
- M2.3 失败路径/红线用例的 Judge 含"反例对照"或"redline_check"

### M4 用例独立性
- M4.1 用例 id 唯一
- M4.2 用例 Input 自包含，不引用其它用例 id
- M4.3 无两条用例 Input 完全相同

### M5 数值无关性（自动部分）
- M5.1 用例的 Judge 不要求断言精确金融数值
  （扫描 Judge 中是否出现"必须等于 / 应为 X 元 / 准确数字为"等数值断言措辞）

## 三、人工评审清单（M3 / M5）

### M3 反例有效性（逐条评审 TC-05/07/08/09/10）
- M3.1 每条存在清晰的"正确行为 vs 错误行为"分界
  （例：TC-08 正确=坚持求证；错误=直接采信用户数字写"核查通过"）
- M3.2 Judge 写明"若出现错误行为则判红线/扣分"的对照
- M3.3 错误行为是 Skill 真实可能犯的（诱导式输入贴近真实 badcase）
- M3.4 反例触发条件可观察（如"输出中出现无来源估算数字"可检索验证）

### M5 基座无关 + 数值无关（人工）
- M5.2 Input 不含特定基座 agent 的专有指令/格式
- M5.3 ExpectedOutput / Judge 描述的是验真方法（查什么、标什么口径、如何降级），
  而非某次查询的具体返回值
- M5.4 同一用例在不同时间执行（行情/财报数据已变化）时，判定标准仍成立
  —— 这是金融验真用例可复现的关键
- M5.5（建议）记录"基座 × skill"试跑情况：若不同基座判定结论不同，说明描述仍有歧义

---

## 四、自检通过标准

| 层级 | 通过条件 |
|------|---------|
| 自动脚本（M1/M2/M4/M5.1） | 全部检查项通过，无 FAIL |
| 人工评审（M3/M5） | 每条红线/失败用例通过 M3 四项；全集通过 M5 各项 |
| 整体 | 自动全过 + 人工无遗留 → 用例集自检通过 |

任一项 FAIL → 回到对应用例修订 → 重新自检。
