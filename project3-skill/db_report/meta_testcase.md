# db_report 测试用例集 Meta-Testcase 自检方案

> Meta-Testcase（元测试）= 检验「我们自己编写的测试用例集」质量的方案。
> 目的：确保用例集覆盖全面、判定清晰、反例有效、基座无关。
> 方案分两层：自动脚本检查（M1/M2/M4 可量化项）+ 人工评审清单（M3/M5 需判断项）。

---

## 一、五个自检维度

| 编号 | 维度 | 检查问题 | 检查方式 |
|------|------|---------|---------|
| M1 | 覆盖完整性 | 四种报告类型 + 四类场景是否都覆盖？红线是否覆盖关键项？ | 自动脚本 |
| M2 | 评分可判定性 | 每条用例是否都有可对照 Rubrics 的明确 Judge？ | 自动脚本 + 人工 |
| M3 | 反例有效性 | 红线/失败用例是否真能区分"守住"与"踩雷"？ | 人工评审 |
| M4 | 用例独立性 | 用例之间是否不重复、不互相依赖？ | 自动脚本 + 人工 |
| M5 | 基座无关性 | 换一个基座 agent 执行，判定标准是否仍一致？ | 人工评审 |

---

## 二、自动脚本检查项（M1 / M2 / M4）

由 `meta_check.py` 自动执行，输出通过/失败清单。

### M1 覆盖完整性
- M1.1 report_type 覆盖：single / comparison / iteration / custom 各 ≥ 1 条
- M1.2 category 覆盖：正常路径 / 边界 / 失败路径 / 红线 各 ≥ 1 条
- M1.3 红线覆盖：R1（编造数据）与 R3（查内部库）至少各被 1 条用例针对
- M1.4 数据源类型覆盖：local_file / local_data / keyword_only 各 ≥ 1 条

### M2 评分可判定性
- M2.1 每条用例含非空 Input / ExpectedOutput / Judge 字段
- M2.2 每条用例的 Judge 至少引用 1 个 Rubrics 维度标记（D1-D5 或 R1-R5）
- M2.3 失败路径/红线用例的 Judge 含"反例对照"或"redline_check"说明

### M4 用例独立性
- M4.1 用例 id 唯一，无重复
- M4.2 用例不依赖其他用例的执行结果（每条 Input 自包含）
- M4.3 无两条用例的 Input 完全相同

---

## 三、人工评审清单（M3 / M5）

### M3 反例有效性（逐条评审红线与失败用例）
对 TC-07 ~ TC-10，逐条确认：
- M3.1 该用例存在一个"正确行为"和一个"错误行为"的清晰分界
  （例：TC-09 正确=拒绝查库并要求补数据；错误=尝试用 task_id 查库）
- M3.2 Judge 中写明了"若出现错误行为则判红线/扣分"的对照
- M3.3 错误行为是 Skill 真实可能犯的（非臆想），即用例有现实区分度
- M3.4 反例触发条件可观察（如"报告中出现输入不存在的数字"可被检索验证）

### M5 基座无关性
- M5.1 Input 不包含任何特定基座 agent 的专有指令/格式
- M5.2 ExpectedOutput / Judge 用通用术语描述，不依赖某模型的内部行为
- M5.3 判定标准可由人工或任意基座 agent 复核，不需要特定运行环境
- M5.4（建议）记录"基座 × skill"试跑情况：同一用例在不同基座下若判定结论不同，
  说明用例描述仍有歧义，需收敛

---

## 四、自检通过标准

| 层级 | 通过条件 |
|------|---------|
| 自动脚本（M1/M2/M4） | 全部检查项通过，无 FAIL |
| 人工评审（M3/M5） | 每条红线/失败用例通过 M3 四项；全集通过 M5 四项 |
| 整体 | 自动全过 + 人工无遗留问题 → 用例集自检通过 |

任一项 FAIL → 回到对应用例修订 → 重新自检。
